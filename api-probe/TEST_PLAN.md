# Test Plan — Telegram Business API (Phase 1)

Цель — экспериментально подтвердить или опровергнуть поведение Business API, от которого
зависит весь продукт. Главные гипотезы (из `../docs/01-research-telegram-api.md`):

- **H1.** `deleted_business_messages` содержит только `business_connection_id`, `chat`,
  `message_ids[]` — **без текста, без инициатора, без времени удаления**.
- **H2.** По событию удаления **невозможно** определить, кто удалил (собеседник / владелец /
  «для обоих»).
- **H3.** Удаление собеседником **«только у себя»** до бота **не доходит**.
- **H4.** Медиа (фото/голос/видео/документ) приходит в `business_message`; после удаления
  восстановить контент можно только из заранее сохранённого архива.

## Условия проведения

- Аккаунт-владелец с **Premium**, бот с **Business Mode**, подключение через «Чат-боты».
- Второй аккаунт-собеседник для E-3…E-9.
- Сервер `npm run dev` запущен, webhook активен (`npm run webhook-info`).
- Перед каждым экспериментом: заметить `update_id` последнего лога, чтобы отделить новые.
- Все выводы фиксировать по **реальным файлам** из `logs/`, а не по ожиданиям.

### Как читать «ожидаемые поля»
Ожидания ниже — это гипотезы для проверки, а **не** факты. Если реальный payload отличается —
это и есть ценный результат; занесите его в итоговую таблицу.

---

## E-1 — Обычное текстовое сообщение

- **Действия:** собеседник (или владелец) пишет текст в разрешённом чате.
- **Ожидаемый webhook:** `business_message`.
- **Ожидаемые поля:** `message.message_id`, `message.business_connection_id`, `message.from`,
  `message.chat`, `message.date`, `message.text`.
- **Что проверить:**
  - приходит ли `business_connection_id` и совпадает ли с активным подключением;
  - есть ли различие для входящего (от собеседника) и исходящего (от владельца) —
    у исходящего может присутствовать `sender_business_bot`;
  - совпадает ли `chat.id` с тем, что придёт позже в событии удаления (ключ связывания).
- **Вывод:** подтверждается ли, что мы получаем полный `Message` и можем его архивировать
  (фундамент всего продукта — «store-on-arrival»).

## E-2 — Редактирование текста

- **Действия:** отправить текст (E-1), затем отредактировать его.
- **Ожидаемый webhook:** `edited_business_message`.
- **Ожидаемые поля:** тот же `message_id`, новый `text`, `edit_date`, `business_connection_id`.
- **Что проверить:**
  - совпадает ли `message_id` с оригиналом (нужно для истории версий);
  - присутствует ли `edit_date`;
  - приходит ли повторный `edited_business_message` при многократном редактировании;
  - не дублируется ли одно и то же редактирование (важно для дедупликации версий).
- **Вывод:** можем ли восстановить полную цепочку версий сообщения.

## E-3 — Удаление сообщения «для всех» (Delete for everyone)

- **Действия:** отправить текст, затем удалить его выбрав **«Удалить у всех»**.
  Провести дважды: (а) удаляет **собеседник**, (б) удаляет **владелец**.
- **Ожидаемый webhook:** `deleted_business_messages`.
- **Ожидаемые поля:** `business_connection_id`, `chat`, `message_ids[]`. **И больше ничего.**
- **Что проверить (ядро исследования):**
  - **H1:** есть ли в payload `text`/контент, `from`/`initiator`/`deleted_by`, `date` удаления
    (анализатор в консоли явно печатает `ABSENT` для этих полей);
  - **H2:** отличается ли payload при удалении собеседником (а) от удаления владельцем (б).
    Сравнить файлы из `logs/` побайтово по структуре;
  - доходит ли `message_id` до нас в том же виде, что был в E-1 (связывание с архивом).
- **Вывод:**
  - Если контент/инициатор/время отсутствуют → **H1 подтверждена**: показывать удалённое
    можно только из своего архива.
  - Если (а) и (б) структурно идентичны → **H2 подтверждена**: атрибуция удаления невозможна
    (критическое ограничение продукта).

## E-4 — Удаление сообщения только у себя (Delete for me)

- **Действия:** собеседник отправляет текст, затем удаляет его выбрав **«Удалить у меня»**
  (только у себя). Копия у владельца остаётся.
  Повторить: **владелец** удаляет «только у себя».
- **Ожидаемый webhook:** гипотеза — при удалении **собеседником** «у себя» **ничего не
  приходит** (копия владельца цела). При удалении **владельцем** «у себя» —
  `deleted_business_messages` приходит.
- **Ожидаемые поля:** если событие есть — как в E-3.
- **Что проверить:**
  - приходит ли хоть какой-то update, когда «у себя» удаляет **собеседник** (ждать 30–60 с);
  - приходит ли update, когда «у себя» удаляет **владелец**.
- **Вывод (H3):**
  - Нет события при удалении собеседником «у себя» → продукт **не видит** такие удаления;
    ловятся только «удалить у всех» и любые удаления, затрагивающие копию владельца.
  - Это ключевой фактор ценности «поймать удалённое собеседником».

## E-5 — Удаление фотографии

- **Действия:** отправить фото → дождаться `business_message` с `photo` → удалить «у всех».
- **Ожидаемый webhook:** `business_message` (с `photo[]`), затем `deleted_business_messages`.
- **Ожидаемые поля:** в сообщении — `photo[]` c `file_id`/`file_unique_id`; в удалении —
  только `message_ids[]`.
- **Что проверить (H4):**
  - до удаления: сохранить `file_id`; проверить, **резолвится ли** он через `getFile`
    **после** удаления (ручной вызов `getFile` с этим `file_id`);
  - несёт ли событие удаления хоть какую-то ссылку на медиа (ожидается — нет).
- **Вывод:** нужно ли скачивать файл **в момент прихода** (если после удаления `getFile`
  перестаёт работать — да, обязательно).

## E-6 — Удаление голосового

- **Действия:** отправить голосовое → `business_message` с `voice` → удалить «у всех».
- **Ожидаемый webhook:** `business_message` (`voice`), затем `deleted_business_messages`.
- **Ожидаемые поля:** `voice.file_id`, `voice.duration`, `voice.mime_type`; в удалении — как E-3.
- **Что проверить:** аналогично E-5 (жизнь `file_id` после удаления), плюс что голосовое
  вообще доставляется через business-канал.
- **Вывод:** подтверждение доставки голосовых и стратегии архивации медиа.

## E-7 — Удаление видео

- **Действия:** отправить видео (`video`) и/или кружок (`video_note`) → удалить «у всех».
- **Ожидаемый webhook:** `business_message` (`video`/`video_note`), затем `deleted_business_messages`.
- **Ожидаемые поля:** `video.file_id`/`video_note.file_id`, размеры/длительность; удаление — как E-3.
- **Что проверить:** доставка обоих подтипов; поведение `getFile` после удаления; размер файла
  (крупные файлы — ограничение Bot API на скачивание ~20 МБ без self-hosted API).
- **Вывод:** ограничения на архивацию крупного видео.

## E-8 — Удаление документа

- **Действия:** отправить документ (`document`) → удалить «у всех».
- **Ожидаемый webhook:** `business_message` (`document`), затем `deleted_business_messages`.
- **Ожидаемые поля:** `document.file_id`, `file_name`, `mime_type`, `file_size`.
- **Что проверить:** доставка; `getFile` после удаления; лимиты размера.
- **Вывод:** применимость стратегии архивации к произвольным файлам.

## E-9 — Несколько удалённых сообщений подряд

- **Действия:** отправить 5–10 сообщений, затем выделить и удалить их **разом** «у всех».
  Отдельно: удалить быстро по одному подряд.
- **Ожидаемый webhook:** один или несколько `deleted_business_messages`.
- **Ожидаемые поля:** `message_ids[]` с несколькими id — либо один update с массивом, либо
  несколько updates.
- **Что проверить:**
  - приходит ли **один** update с массивом id или **несколько** отдельных;
  - сохраняется ли порядок; нет ли потерь/дублей id;
  - как это влияет на дедупликацию и группировку уведомлений (batch vs по одному).
- **Вывод:** как обрабатывать пакетные удаления (idempotency + группировка).

## E-10 — Отключение Business Connection

- **Действия:** владелец отключает бота в **Настройки → Telegram Бизнес → Чат-боты**
  (или меняет права/набор чатов).
- **Ожидаемый webhook:** `business_connection` с `is_enabled: false` (отключение) или
  обновлёнными `rights` (изменение прав).
- **Ожидаемые поля:** `id`, `user`, `user_chat_id`, `date`, `is_enabled`, `rights`.
- **Что проверить:**
  - приходит ли явное событие отключения и как оно выглядит;
  - совпадает ли `id` подключения с тем, что был при подключении;
  - что приходит при **изменении набора чатов/прав** (не полное отключение);
  - перестают ли после отключения приходить `business_message`.
- **Вывод:** можем ли надёжно детектировать отзыв доступа (нужно для остановки архивации и
  запуска retention/удаления данных).
